# Emit structured TOC manifest so the agent never needs to read the full article for orientation

## Context

Closely related to the chunking suggestion, but independently valuable even without splitting into files. The core insight: the agent's first 5 reads of the article were purely *orientation* — figuring out what's in the article, how it's structured, and where the problems are. The agent didn't start doing useful work until read #6. That's ~12k tokens of pure waste.

## Gap

h2md's current output summary is:

```text
words: 15092
issues: 68
lint_remaining: 419
```

Three numbers. The agent learns nothing about structure, content distribution, or where problems concentrate. It has to read the article to discover that information.

## Suggestion

Add a `toc.json` (or section in `h2md.json`) emitted during the convert stage:

```json
{
  "toc": [
    {
      "level": 2,
      "title": "Full-stack JavaScript runtime",
      "anchor": "full-stack-javascript-runtime",
      "line_start": 22,
      "line_end": 518,
      "word_count": 3200,
      "code_blocks": 18,
      "code_languages": {"javascript": 12, "bash": 4, "text": 2},
      "issues": 12,
      "subsections": [
        {"level": 3, "title": "Frontend Development", "line_start": 37, "line_end": 85, "code_blocks": 2},
        {"level": 3, "title": "Hot Reloading", "line_start": 85, "line_end": 95, "code_blocks": 0},
        ...
      ]
    },
    {
      "level": 2,
      "title": "Bug Fixes",
      "line_start": 2080,
      "line_end": 2500,
      "word_count": 4100,
      "code_blocks": 0,
      "issues": 0
    }
  ],
  "summary": {
    "total_sections": 42,
    "deepest_level": 4,
    "sections_with_issues": 8,
    "largest_section": {"title": "Bug Fixes", "words": 4100},
    "code_heavy_sections": ["Full-stack JavaScript runtime", "Package Management", "Testing"]
  }
}
```

### Why this matters beyond just navigation

1. **The agent can skip clean sections entirely.** "Bug Fixes" has 4100 words but zero code blocks and zero issues — no reason to read it. That's ~5k tokens saved by not reading one section.

2. **The agent can prioritize.** "Full-stack JavaScript runtime" has 18 code blocks and 12 issues — that's where the work is. Read that first.

3. **The agent can estimate work.** 8 sections with issues, ~3 issues each, most in code blocks → "this needs a batch code-fix script, not manual edits." That strategic decision happens at ~200 tokens instead of ~12k.

4. **Code language distribution reveals systematic problems.** If toc.json shows 40 blocks tagged `python` on a JavaScript blog, the agent knows immediately that language detection failed globally — write a batch fix, don't edit one by one.

5. **Line ranges enable surgical reads.** Instead of `Read(offset=0, limit=120)` and hoping to hit something interesting, the agent can `Read(offset=37, limit=48)` to read exactly the "Frontend Development" section.

### Interaction with --chunk

If `--chunk` splits the file, each chunk file is small enough to read whole. The toc.json still helps: the agent reads it first to decide *which* chunks need attention, skipping clean ones entirely.

If the article stays monolithic, toc.json provides the same skip/prioritize/target benefits without any file-splitting overhead.

Either way: the agent's first action after h2md is `Read toc.json`, not `Read article.md lines 1-120`.

## Impact

Eliminates the orientation phase entirely. The agent goes from "read 13 sequential chunks to understand the article" to "read 1 manifest file and know exactly where to focus." Saves ~12k tokens on large articles and makes the agent's fix strategy optimal from the first action.
