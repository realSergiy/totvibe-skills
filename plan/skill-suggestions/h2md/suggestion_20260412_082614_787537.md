# Site-profile system for recurring domain-specific extraction rules

## Context

Tech blogs (bun.com, deno.com, vercel.com, astro.build) share common patterns: Shiki highlighting, tabbed code blocks, clipboard copy buttons, similar CSS class conventions. Every conversion from these sites will hit the same issues. The agent workarounds from this session (Python spacing script, tab merging, clipboard stripping) are reusable knowledge that will be re-discovered and re-implemented on every future conversion from these sites.

## Gap

h2md treats every site as a blank slate. There's no way to encode "bun.com uses Shiki + CodeTabs + CopyIcon" so that future conversions just work.

## Suggestion

Add a site-profile system — small config files that encode domain-specific extraction rules:

```yaml
# ~/.config/h2md/profiles/bun.com.yaml
domain: bun.com
code_blocks:
  syntax_highlighter: shiki
  strip_elements: [".CopyIcon", ".clipboard"]
  language_hints:
    ".CodeBlockShell": bash
    ".CodeBlockPowershell": powershell
    default: javascript
tabs:
  container: ".CodeTabs"
  header: ".CodeTabsHeader .CodeTab"
  panel: ".CodeTabItem"
labels:
  file_labels: ".CodeBlockTab"  # e.g., "bunfig.toml", "package.json"
```

**Discovery**: h2md could auto-generate a draft profile on first conversion of a domain, based on detected patterns (Shiki classes, tab structures, copy buttons). The agent or user reviews and saves it.

**Sharing**: Profiles could be distributed as a community repository (like Homebrew taps or ESLint configs). `h2md --install-profile bun.com` pulls the config.

**Fallback**: Without a profile, h2md behaves exactly as today. Profiles are pure additive.

## Impact

Converts the one-time cost of discovering site-specific patterns into reusable knowledge. Second conversion from the same domain would need zero agent post-processing. Over time, the community profiles would cover the most common documentation sites.
