# Chunked output mode: split large articles by heading into separate files with an index

## Context

Converting <https://bun.com/blog/bun-v1.3> — a 15,000-word, 2,574-line article with 42 sections. The agent had to read it in 5 sequential chunks (offset 0-120, 120-240, 240-360... up to 2574) just to assess quality, burning ~25k tokens on reads alone. Each Read call returned content the agent had to hold in context while planning fixes. The article was too large for the agent to "see" all at once, so it operated partially blind — fixing section A, then discovering section K had the same issue, then going back.

The fundamental problem: h2md produces one monolithic file, but the agent consuming it has a finite context window and reads in serial. A 15k-word article is simply too large to process efficiently as a single unit.

## Gap

The agent's workflow was:

1. Read lines 1-120 → spot spacing issues, wrong language tags
2. Read lines 120-240 → same issues, plus tab UI artifacts
3. Read lines 240-360 → same pattern continues
4. Read lines 360-560, 560-760, 760-960, 960-1160, 1160-1460, 1460-1760, 1760-2060, 2060-2360, 2360-2574 → 13 total reads
5. Write a Python script to batch-fix systematic issues across the whole file
6. Run 15 manual edits, re-reading sections to verify

This serial-read-then-global-fix pattern is inherently wasteful. If the article were split into self-contained chapters, the agent could:

- Process each chapter independently (or in parallel via subagents)
- Hold an entire chapter in context at once
- Apply chapter-specific fixes without cross-referencing distant parts of the file

## Responsibility

The tool knows the heading structure at extraction time. It has all the information needed to split intelligently. The agent discovers this structure only after expensive sequential reads.

## Suggestion

### `--chunk` flag: split output by heading level

```bash
h2md https://bun.com/blog/bun-v1.3 --chunk=h2
```

This produces a workspace like:

```text
article/
  _index.md              # frontmatter + intro (before first h2) + TOC with links
  01_full-stack-javascript-runtime.md
  02_bundler-build.md
  03_package-management.md
  04_testing-and-debugging.md
  05_apis-standards.md
  06_security-enhancements.md
  07_node-js-compatibility.md
  08_developer-experience.md
  09_utilities.md
  10_processes-shell.md
  11_performance-improvements.md
  12_bug-fixes.md
  13_breaking-changes.md
  article.md             # full concatenated file (still available)
```

Each chunk file is self-contained:

- Has its own frontmatter (`title`, `parent`, `section_number`)
- Contains the heading and all content until the next heading at the same level
- Small enough to read in a single Read call (~200-400 lines each)
- Can be edited independently

The `_index.md` file contains:

- The article's top-level frontmatter
- Any content before the first split heading (intro, install block)
- A table of contents linking to each chunk
- Per-chunk metadata: word count, code block count, issue count

### Why this changes the agent workflow fundamentally

**Before (monolithic):**

```text
Read 13 chunks serially → hold everything in context → write global fix script → apply → verify
Total: ~25k read tokens + ~15k edit tokens = ~40k tokens
```

**After (chunked):**

```text
Read _index.md (200 lines) → see which chapters need attention → read only those → fix each independently
Total: ~3k read tokens + ~5k edit tokens = ~8k tokens
```

Even better, the agent can parallelize: spawn subagents to fix chapters 1, 3, 5 concurrently while processing 2, 4, 6 itself.

### `--chunk` interacts well with other suggestions

- **With `--auto-fix`**: auto-fix runs per-chunk, so chapter-specific issues (e.g., one chapter has TOML blocks, another has bash) get appropriate treatment.
- **With notes.md**: issues are grouped by chunk file with line numbers relative to that chunk, not the 2574-line monolith.
- **With `--copy-to`**: copies the final concatenated file (or the directory).

### The `--chunk=auto` heuristic

When the agent doesn't know the article structure upfront:

```bash
h2md https://example.com/post --chunk=auto
```

Auto mode: split if article > 1000 words. Use the first heading level that produces 3-15 chunks. For this article, h2 produces 13 chunks (ideal). For a shorter article with many h2s, it might choose h3 or not split at all.

### Reassembly

The chunks can always be concatenated back:

```bash
cat article/[0-9]*.md > article/assembled.md
```

Or h2md could provide a `--reassemble` flag that concatenates chunks, strips per-chunk frontmatter, and restores the unified article.

## Impact

For articles over ~1000 words:

- **5x fewer read tokens**: agent reads index + targeted chapters instead of the whole file
- **Enables parallel processing**: independent chapters can be fixed by concurrent subagents
- **Better agent accuracy**: each chapter fits in a single Read call, so the agent sees complete context without pagination
- **Composable output**: chapters can be consumed independently (e.g., user only wants the "Bug Fixes" section)

For this specific session: would have reduced the 13 sequential reads (~25k tokens) to ~3 targeted reads (~5k tokens), and enabled parallel editing of chapters instead of a serial Python-script-then-manual-edits approach.
