# Shiki-highlighted code blocks lose formatting: token spacing, wrong languages, UI script leakage

## Context

Converting <https://bun.com/blog/bun-v1.3> — a 15k-word blog post heavy with code examples. The site uses Shiki syntax highlighting which wraps every token in individual `<span>` elements with inter-element whitespace.

## Gap

Three classes of code block issues required ~200 lines of post-processing Python to fix:

**1. Shiki token spacing.** The HTML renders code as `<span>"</span> <span>bun</span> <span>"</span>` which h2md extracts as `" bun "` instead of `"bun"`. Similarly `Response. json` instead of `Response.json`, `). toBe` instead of `).toBe`, `;` instead of `;`, etc. This affected 207 lines across the entire document.

**2. Wrong code fence languages.** All JavaScript/TypeScript code blocks were tagged as ` ```python ` or ` ```text `. The source HTML uses CSS classes like `CodeBlock` and Shiki's `class="shiki"` but the actual language isn't in a `<code class="language-X">` attribute — it's in a sibling `<div class="CodeBlockTab">` element. h2md picked the wrong language or fell back to a generic one.

**3. Interactive UI script leakage.** Copy-to-clipboard `<script>` elements embedded inside code block containers were extracted as code content. The terminal output block for `bun './**/*.html'` included an entire `document.addEventListener("DOMContentLoaded", ...)` handler concatenated with the actual CLI output.

**4. Tabbed code blocks merged.** The install section uses a `CodeTabs` component with tabs for curl/powershell/npm/brew/docker. Each tab's content was concatenated together with tab labels appearing as bare words between code fences, producing a confusing jumble.

## Responsibility

These are all deterministic structural patterns in the source HTML that h2md's extraction pipeline should handle. The workaround I wrote was a two-pass Python script that:

- Split the markdown on code fences
- Applied regex fixes only inside code blocks (collapsing Shiki token spaces, fixing `). method` patterns, removing stray `;` spaces)
- Replaced `python` fence language with `javascript`

This is exactly the kind of structural preprocessing that belongs in h2md's `article.html → article.raw.md` conversion stage, not in the caller's post-processing.

## Suggestion

1. **Shiki-aware text extraction.** When extracting text from `<pre><code>` blocks that contain Shiki-style `<span>` markup, collapse inter-element whitespace. The heuristic: if a `<span>` ends with a non-whitespace character and the next `<span>` starts with a non-whitespace character, don't insert whitespace between them. This is the single highest-impact fix — it affected every code block in the document.

2. **Language detection from sibling elements.** Look for language hints in sibling/parent elements like `<div class="CodeBlockTab">javascript</div>` or class names containing language identifiers (e.g., `CodeBlockShell` → `bash`). Fall back to content-based heuristic (import/export/const → javascript, SELECT/INSERT → sql) rather than defaulting to `python` or `text`.

3. **Script tag filtering in code containers.** Strip `<script>` elements found inside code block containers during the structural preprocessing phase (before text extraction). These are page interactivity scripts, not code content.

4. **Tabbed code block support.** Detect `CodeTabs`-style containers (multiple code blocks with tab labels) and emit them as separate fenced blocks with a bold label prefix, e.g. `**curl:**` followed by the code fence. This is a common pattern on documentation sites (Docusaurus, Nextra, Bun's site, etc.).

## Impact

Eliminates the most time-consuming manual post-processing step for sites using Shiki (which is extremely common — used by VitePress, Astro, Docusaurus, Nuxt Content, and most modern doc/blog platforms). Would have saved ~15 minutes of Python scripting and manual verification on this 15k-word article.
