# Skill prompt agent workflow section is optimized for small articles, burns tokens on large ones

## Context

The h2md skill prompt instructs the agent to: "Read notes.md first → Edit article.md → Cross-reference article.html." For a 15k-word article (2574 lines), this meant 5+ Read calls just to see the article (~25k tokens), plus reading notes.md (275 lines, ~3k tokens), and the agent still had to write a Python script because the issues were too systematic for manual editing.

## Gap

The skill prompt assumes a small article with a handful of surgical fixes. It doesn't account for:

- Articles over ~500 lines where reading the full file is expensive
- Systematic issues affecting every code block (not amenable to individual edits)
- The total token budget — 101k tokens for what should be a ~10k token operation

The prompt also references `--save` as a flag, but h2md doesn't support it.

## Responsibility

The skill prompt is the agent's only guidance. If it sends the agent down a token-expensive path for large articles, that's a skill-level issue.

## Suggestion

1. **Size-aware workflow guidance.** Add a conditional in the skill prompt:

   ```text
   If word_count > 3000:
     - Read only the summary/stats, not the full article
     - Check if issues > 10: likely systematic, consider batch fix
     - Skip notes.md if issues are mostly "fused text" (usually false positives)
   ```

2. **Emit a machine-readable issue manifest** alongside notes.md — a JSON file with line numbers, issue types, and suggested fixes. The agent can process this programmatically instead of parsing prose.

3. **Remove the `--save` reference** from the skill prompt (it doesn't exist) or implement it.

4. **Add a "quality score"** to h2md output: 0-100 based on (untagged code blocks, detected artifacts, lint violations). If score > 90, the agent can skip the review loop entirely. For this article, the score would have been ~30, signaling heavy post-processing needed.

## Impact

Prevents the agent from burning tokens on a read-everything-then-fix-everything loop when the tool should have flagged "this needs batch processing" upfront. Could cut the large-article workflow from 100k tokens to 20k.
