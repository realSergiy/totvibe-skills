# Add a "Recommendation" interaction type for when the assistant has strong domain analysis to bring

## Context

Just finished a six-round /plan-storm session designing a non-trivial skill+CLI in this monorepo. Several interactions across rounds 4 and 5 were laid out as "here are 3 options + write-your-own + tradeoffs" — and the user replied "1: 1, 2: 1, 3: 1", taking my recommendations straight. The single highest-leverage interaction in the whole session was when the user asked "which of these is better?" and I responded with a structured **recommendation** that named the pick, gave reasoning, and flagged what would change my mind. That one interaction probably collapsed two rounds.

## Gap

Plan-storm's interaction-quality guidance pushes toward **neutrality** — name the tradeoffs, include a write-your-own, never accept a contradicting answer silently. This is right for genuine *exploration* moments, when the design space is wide open and the user's domain knowledge should drive.

But for design rounds where the assistant has done structured analysis the user hasn't (architectural patterns, prior-art tradeoffs, technical-feasibility constraints), the protocol implicitly discourages putting opinions on the table. The three sanctioned interaction types are Clarification (asking for missing info), Creative idea (offering options), and Challenge (pushing back on user picks). There is no symmetrical "Recommendation" type for *putting forward a strong assistant pick with reasoning*.

The result: rounds where the assistant could lead with a verdict end up framed as "here are 4 options, you pick", and the user spends mental cycles re-deriving an analysis the assistant already did silently before drafting the round.

## Responsibility

Brainstorming is a partnership. When one partner has strong relevant analysis, holding it back to be neutral wastes the other's time. The protocol should bless directness when warranted — not just neutrality.

## Suggestion

Add a fourth interaction type: **Recommendation** — used when the assistant has done analysis the user hasn't and a clear pick with reasoning is more valuable than a balanced menu.

Format:

```markdown
### Interaction <n> — Recommendation: <one-line topic>

**My pick: <option>** — <one paragraph laying out the reasoning, including what would change my mind>.

Alternatives considered:
1. **<alt>** — why it's worse: <one line>.
2. **<alt>** — why it's worse: <one line>.

If you'd rather a different direction, reply with the number or write your own.
```

The "what would change my mind" framing is load-bearing: it gives the user explicit handles to push back without the assistant feeling defensive about its recommendation.

**When to use Recommendation:** assistant has technical/architectural expertise that's load-bearing; prior art is clear; user has just asked "which is better?".

**When to use Creative idea (existing):** design space is genuinely open and user's domain knowledge should drive.

Add to the "What makes options brilliant, not lazy" section so authors know the choice is principled, not lazy.

## Impact

Cuts back-and-forth in design-heavy sessions. In my session, ~2 rounds of "here are options" → "1: 1" → "OK locking in" could have been one round of "Recommendation: X for these reasons; locking unless you object". Same outcome, half the turns.
