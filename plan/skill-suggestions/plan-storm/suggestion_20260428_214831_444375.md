# Recognize architectural pivots as a distinct event, not silent absorption

## Context

In round 2 of a /plan-storm session, the user added: "the new skill should call /plan-storm under the hood". This wasn't a refinement of the round-1 plan — it was a fundamental architectural pivot that turned the design from "thin CLI" into "plan-storm wrapper + thin CLI". I correctly noted readiness dropped 5% and continued the round loop with new interactions, treating it as a normal refinement.

In retrospect I should have stopped, said "this is a pivot, not a refinement — re-anchoring with fresh round-1 questions", and asked broader questions before continuing. Instead, the next 1–2 rounds were spent grafting the new constraint onto stale architecture.

## Gap

Plan-storm's round loop is uniform: re-read plan, absorb input, update plan, score, present interactions. The protocol acknowledges that "readiness can go down — that's healthy" but treats *all* input as fitting into the existing structure. There is no mechanism for recognizing that *the structure itself* needs to change.

The result: when a pivot happens, the assistant continues the loop with the new constraint awkwardly grafted onto the old plan, and the next 1–2 rounds reconcile structures that should have been redrawn from scratch.

## Responsibility

Recognizing when "we're not refining anymore, we're re-architecting" is a key skill of a brainstorming partner. Silently absorbing a pivot and grinding on with the existing plan structure is a worse experience than naming the pivot and resetting the frame.

## Suggestion

Add explicit pivot-detection guidance to the round loop:

> **Pivot detection.** When absorbing the user's reply, check if it:
>
> - introduces a load-bearing dependency or composition that wasn't in the previous plan,
> - inverts a `[DECIDED]` architectural choice,
> - changes the project from "self-contained" to "wrapper / orchestrator / integration",
> - shifts the primary user or consumer.
>
> If yes, **call it out explicitly**: tell the user "this is a pivot, not a refinement — I'm restructuring the plan back toward round-1 shape." Drop readiness substantially (e.g. −15% to −25%), do a partial-rewrite of the plan rather than an in-place edit, and use the next round's interactions to probe the new shape (more like round-1 questions than convergence questions).

Three benefits: (1) signals to the user that prior rounds' detail won't all carry forward, (2) gives the assistant license to do a bigger plan rewrite, (3) avoids the awkward stage of grafting new constraints onto stale architecture.

## Impact

In my session, naming the round-2 pivot would have telescoped 1–2 follow-up rounds — instead of grafting the plan-storm composition onto the existing CLI-centric draft, I'd have re-drawn the architecture from scratch with the new shape, and several round-3 questions wouldn't have been needed at all.
