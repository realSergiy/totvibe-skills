# peek should support a Python expression mode for transforms SQL can't handle

## Context

Same player-matching analysis session. After the SQL `regexp_replace` failure, I needed to run 5+ increasingly complex data explorations: computing slugified player codes, testing fuzzy matching with `difflib`, grouping by computed columns. Each time I wrote a full Python script (~15-30 lines) with `import polars`, `pl.read_parquet(...)`, processing, and print formatting.

## Gap

Peek handles "read file, inspect it" brilliantly. But the moment you need a **computed column** or a **Python-native transform** (regex, fuzzy match, custom function), you fall off a cliff into full boilerplate. There's no middle ground between `peek -q "SELECT ..."` (limited to Polars SQL) and a raw Python script.

Example of what I wanted to express but couldn't:

```bash
peek kaggle.parquet sackmann.parquet --expr "
t1.with_columns(
    pl.col('player_code').str.replace(r'\d+$', '').alias('base_code')
).join(t2, left_on='base_code', right_on='player_code').select('player_code', 'base_code')
"
```

Instead I wrote:

```python
import polars as pl
k = pl.read_parquet("data/prep/kaggle_players.parquet")
s = pl.read_parquet("data/prep/sackmann_players.parquet")
s_codes = set(s["player_code"].to_list())
# ... 20 more lines of setup, loops, formatting ...
```

## Responsibility

Peek already registers DataFrames as `t1`, `t2` for SQL. Exposing them to a Python expression (or a mini Polars-native DSL) would cover the gap between "simple SQL" and "full script" — which is exactly where most interactive data exploration lives.

## Suggestion

Add a `--expr` / `-e` mode that evaluates a Python expression with `t` / `t1`, `t2` pre-loaded as Polars DataFrames, and prints the result through the existing TOON encoder. Something like:

```bash
peek a.parquet b.parquet -e "t1.filter(~pl.col('code').is_in(t2['code'])).height"
peek a.parquet -e "t.with_columns(pl.col('name').str.to_lowercase()).unique('name').height"
```

## Impact

Eliminates the "write a throwaway Python script" escape hatch for ~80% of exploratory queries that go beyond SQL. Keeps the peek workflow (short command, TOON output, no boilerplate) intact for the cases where SQL falls short.
