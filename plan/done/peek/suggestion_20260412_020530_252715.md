# SQL mode silently promises full SQL but fails on common string functions

## Context

Analyzing 100 unmatched player codes between two parquet files (`kaggle_players.parquet` and `sackmann_players.parquet`). I needed to test matching heuristics by transforming `player_code` strings — stripping trailing digits, removing secondary initials, truncating compound lastnames — and then re-joining against the other file.

## Gap

I tried a multi-file SQL query with `regexp_replace`:

```bash
peek kaggle_players.parquet sackmann_players.parquet -q "
WITH h1 AS (
  SELECT player_code, regexp_replace(player_code, '[0-9]+$', '') as candidate FROM t1
)
SELECT COUNT(*) FROM h1 JOIN t2 ON h1.candidate = t2.player_code
"
```

This failed with `SQLInterfaceError: unsupported function 'regexp_replace'`. Polars SQL only supports a subset of SQL functions, but peek's docs don't mention this limitation. I had to fall back to a full `uv run python` script with Polars native API — writing import boilerplate, reading both files manually, and doing the joins in Python. This happened multiple times during the session for progressively more complex heuristic tests.

## Responsibility

Peek's SQL mode is positioned as "arbitrary SQL" in the docs. When a user hits an unsupported function, the error is opaque (comes from Polars internals, not peek). Peek should either:

- Document which SQL functions are available (or link to Polars SQL docs)
- Catch the error and suggest a Polars-native alternative
- Or ideally, support common string functions by translating them to Polars expressions under the hood

## Suggestion

At minimum: add a note to the skill docs listing common **unsupported** SQL functions (regex functions, `regexp_replace`, `regexp_matches`, `COALESCE` edge cases) so the caller knows upfront not to attempt them. Even better: intercept `SQLInterfaceError` and print a one-line hint like "regexp_replace is not available in Polars SQL — try a Python one-liner with `pl.col(...).str.replace(...)`".

## Impact

Saves a round-trip failure + workaround per query. In this session I hit this wall once and then preemptively avoided peek's SQL mode for all subsequent analysis, falling back to verbose Python scripts (~20 lines each) for what could have been one-liner queries — if I'd known the boundary upfront.
