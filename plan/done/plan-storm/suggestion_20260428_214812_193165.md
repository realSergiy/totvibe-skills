# Add a round-1 context probe for integration points and adjacent systems

## Context

In a six-round /plan-storm session designing a skill in this monorepo, several major architectural pivots happened in rounds 2–5 because of *latent requirements* the user hadn't surfaced in their initial brief:

- Round 2: "the new skill should call /plan-storm under the hood" (massive — flipped from thin CLI to plan-storm wrapper).
- Round 3: "later there will be an auto-implementing agent on every repo" (changed submission shape from gist to PR+branch).
- Round 4: "CLI must be usable standalone with hand-rolled files" (simplified architecture).
- Round 5: "drop autocomplete; push everything into the CLI; minimize SKILL.md".

Each pivot required substantial plan rework. They were predictable in retrospect — every one was about how the new thing relates to *adjacent things* (other skills, future consumers, alternative invocation paths, repo-wide architecture).

## Gap

Plan-storm's round-1 captures Vision, Problem, Users, Goals — all *internal* to the project being planned. It doesn't probe for *integration context*: what other systems does this interact with? Who or what consumes it besides the direct invoker? What are alternative invocation paths? What constraints come from the surrounding ecosystem?

For self-contained projects (a CLI utility, a UI component) this is fine. For tools that compose with other tools — which is most modern software work, and definitely most skills in a skill monorepo — integration context is load-bearing and should surface architectural decisions early. Plan-storm's current round-1 actively drives the conversation *inward* (vision, users, scenarios) when the load-bearing questions are *outward*.

## Responsibility

Plan-storm's round-1 sets the trajectory for the entire session. If foundational integration questions aren't asked in round 1, they emerge as pivots in rounds 3–5, and prior rounds' work partially gets thrown away. The protocol should make the integration probe a default, especially when the project lives in a context the assistant can detect (e.g. inside an existing repo with related code).

## Suggestion

Add to the "Round 1 specifically" guidance: **always include at least one Clarification interaction targeting integration context.** Sample prompts:

- "What other tools / services / skills does this need to compose with — at runtime or via hand-off?"
- "Who or what consumes this besides the user invoking it directly? Future agents, scheduled jobs, other team members?"
- "Are there alternative invocation paths or escape hatches we should preserve (e.g. manual use without the surrounding orchestration)?"
- "What in the surrounding system constrains this — auth model, naming conventions, deployment targets, peer projects?"

When the assistant can see the surrounding repo (e.g. SKILL.md is being drafted inside a skill monorepo), the protocol should also direct it to **proactively scan for related code/skills before round 1** and surface anything relevant in the framing.

Pair with the existing Constraints section — integration findings get logged in §6, not invented inside the project bounds.

## Impact

Catches latent requirements at round 1, when the plan is still cheap to reshape. In my session, three of the four big pivots were integration-context: composition with /plan-storm, the future-agent consumer, the standalone-CLI escape hatch. Asking these in round 1 would have collapsed at least 2 rounds of re-architecture.
