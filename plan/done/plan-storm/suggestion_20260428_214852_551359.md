# Defer slug picking when the user's brief contains a working name they may not be committed to

## Context

In a /plan-storm session for what eventually became "stormitem", round 1 picked the name "workitem" from the user's brief (where they said "create a new workitem skill+cli"). Round 1.5, the user said "let's call it stormitem instead. Please rename." That consumed an entire conversational turn for an administrative rename. The user clearly hadn't decided on the name when they wrote the brief — they were waiting to see the shape before committing.

## Gap

Plan-storm's round-1 protocol picks the slug from the user's opening message: *"If the user's opening message contains an obvious noun phrase for the thing being built, turn that into a 1–3 word kebab-case slug."* This works when the user has a name in mind. It fails for the very common case where the brief contains a *working term* but the user is open to renaming once design clarifies.

The fallback is also weak: "If you genuinely can't tell what to call it, pick a placeholder like `plan/draft.md`." This frames placeholders as the option-of-last-resort, when in practice they're the *right* default whenever the brief's name was working-language rather than committed branding.

## Responsibility

Naming benefits from having the design partially in view. Forcing the name decision at round 1 — when readiness is 15% — produces churn when the user later wants to rename. The cost isn't the rename itself (which is cheap: `mv` + `replace_all`); it's the conversational round it consumes, which in long sessions adds up.

## Suggestion

Two small changes:

1. **Reframe the placeholder as a confident default**, not a fallback. If the user's brief uses a working term that wasn't presented as a name choice ("create a new X skill" rather than "call this `X`"), default to a placeholder slug (`plan/draft.md`, or `plan/draft-N.md` if it exists) and surface naming as an early interaction rather than auto-picking.

2. **Add a naming Clarification at round 1 or 2** when a placeholder is in use. Format:

   > ### Interaction n — Clarification: name?
   > Working name in your brief: `<term>`. Suggestions: `foo`, `bar`, `baz`. Or write your own. (Defer if you'd rather see more shape first — we can rename anytime.)

   The "we can rename anytime" reassurance is important: it lets the user commit lightly knowing it's reversible.

3. **In the round-1 message, mention rename-cost up front**: *"Plan saved to `<path>`. Rename anytime — just say so."* Currently the protocol says "If they push back, rename the file" but doesn't reassure the user proactively, so users default to mentioning naming concerns *late*, after the file already has content tied to it.

## Impact

Saves a turn whenever the name evolves between brief and design. Each turn saved is also a cache-warm continuation of the conversation, which compounds across long sessions.
