# Edge labels auto-wrap at unfortunate boundaries — skill should hint at `<br/>`

## Context
Drawing a flowchart-based architecture diagram for a 5-service system. Used `flowchart LR` with a subgraph for grouping and labeled bidirectional edges to communicate the kind of each interaction (HTTP, atomic INSERT + queue send, OCR call, etc.).

## Gap
I produced descriptive edge labels in single-line form, e.g.:

```text
TS -->|"1. stream PDF (PUT source.pdf)"| S3
TS <-->|"3. proxy page reads (GET pages/n.md)"| S3
```

The renderer auto-wrapped them at word boundaries at unfortunate places:

```text
1. stream PDF (PUT
source.pdf)

3. proxy page reads (GET
pages/n.md)
```

The user noticed the ugliness and corrected me, pointing out that `<br/>` works in edge labels just like in node labels. After hand-placing breaks (`PUT source.pdf<br/>GET pages/n.md`) the diagram became readable.

## Responsibility
The skill maintains a "Common pitfalls" section precisely to head off predictable failure modes. Long edge labels getting auto-wrapped at random word boundaries is one of those failures — I'd never have shipped the bad version if the skill had flagged it. Currently the only label-shape pitfall, "Special characters in labels", only covers escaping `()`/`:`/`[]` and HTML entities; it doesn't mention label length or `<br/>`. The minimal Flowchart template uses only one-word edge labels (`Yes`, `No`), so the pattern isn't modeled by example either.

## Suggestion
Reword the existing "Special characters in labels" pitfall (don't add a new entry) so it covers length too:

> **Labels: special chars and length.** Wrap labels containing parentheses, colons, or square brackets in `"..."`. Use `<br/>` to force line breaks in long labels — *especially edge labels*, which renderers otherwise auto-wrap at unfortunate boundaries. HTML entities (`#quot;`, `#35;`) work for the trickiest characters.

One sentence added, entry stays roughly the same length. Inserts the load-bearing fact (edge labels auto-wrap; `<br/>` is the fix) into the place a reader will already check when working with labels.

**Optional alternative (cheaper, train-by-example):** in the minimal Flowchart template, change one edge label to demonstrate the pattern, e.g. `B -->|"Yes,<br/>proceed"| C`. Adds zero prose. Either change alone would have caught me.

## Impact
Saves diagrams from the "looks fine in the source, looks ugly when rendered" trap. The fix is a one-line skill edit; the bug it prevents is the kind that requires a round-trip with the user to fix.
