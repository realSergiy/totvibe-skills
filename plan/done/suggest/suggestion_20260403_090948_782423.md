# Proactive trigger language is too weak — I ignored it 3 times

## Context

During a merge-players task, I needed cross-file parquet comparison. peek couldn't do it, so I wrote 3 separate `uv run python3 -c "import polars..."` workarounds. The suggest skill description says "Use PROACTIVELY whenever you hit a skill gap, write a workaround..." — but I didn't submit a suggestion until the user explicitly asked me to.

## Gap

The proactive instruction exists but didn't trigger action. I read the skill list at the start of the conversation, saw "Use PROACTIVELY whenever you hit a skill gap, write a workaround, or notice verbose/redundant output" in the suggest skill's one-liner, and still wrote 3 workarounds without suggesting. The instruction is:

- Buried in the middle of the skill description
- Phrased as guidance ("Use PROACTIVELY") rather than a requirement
- Not reinforced at the point where I'm actually deciding whether to use a skill

## Responsibility

The suggest skill's description is the only place that can tell me when to use it. If the trigger language isn't strong enough to interrupt my workflow, suggestions only happen when users remind me — defeating the purpose.

## Suggestion

Strengthen the trigger in the skill's one-liner description (the part shown in the system-reminder skill list) to use BLOCKING REQUIREMENT language, similar to how skill invocation is described. Something like:

Current: `"Use PROACTIVELY whenever you hit a skill gap, write a workaround, or notice verbose/redundant output."`

Proposed: `"BLOCKING REQUIREMENT: When you write a workaround for a skill limitation (python one-liner, extra parsing, manual post-processing), you MUST submit a /suggest IMMEDIATELY — before continuing the task. Do not batch suggestions for later."`

The word "IMMEDIATELY" and "MUST" match the patterns that actually interrupt my flow (like "IMPORTANT: these instructions OVERRIDE..." or "This is a BLOCKING REQUIREMENT: invoke the relevant Skill tool BEFORE generating any other response").

## Impact

Turns suggest from a feature I know about but forget to use into one that fires reliably at the moment the gap is fresh — when the workaround code is right there to include as evidence.
